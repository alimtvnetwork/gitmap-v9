# Golden Fixture Regeneration — Safe Workflow

## Overview

`gitmap` pins on-disk output (CSV reports, JSON reports, dry-run plans,
shell completions, scan artifacts) with **golden fixture tests**.
These tests compare byte-for-byte against checked-in files under
`testdata/` and fail CI on any drift.

When you **intentionally** change formatting (a new field, a renamed
column, a schema-version bump, a CSV quoting tweak), the golden
fixtures must be regenerated and committed alongside the code change.
This document is the canonical procedure.

> **Audience**: maintainers landing a deliberate format change.
> **CI must never set `GITMAP_ALLOW_GOLDEN_UPDATE`.** Regeneration
> is a **local, human-driven** step.

---

## 1. The Two-Key Safety Gate

Every golden test guards regeneration with **two** environment
variables. Both must be set to `1` before any fixture is overwritten.

| Env var                       | Role                                  | Set by      |
|-------------------------------|---------------------------------------|-------------|
| `GITMAP_UPDATE_GOLDEN`        | Per-test trigger ("please regenerate")| Developer   |
| `GITMAP_ALLOW_GOLDEN_UPDATE`  | Global allow-gate (defense in depth)  | Developer   |

The gate is centralized in `gitmap/goldenguard/goldenguard.go`. If
`GITMAP_UPDATE_GOLDEN=1` is set but `GITMAP_ALLOW_GOLDEN_UPDATE=1` is
**not**, the test calls `t.Fatalf` with a clear remediation message —
silent skipping would let a misconfigured CI quietly regenerate.

**Why two keys?** A stray `GITMAP_UPDATE_GOLDEN=1` in a developer's
shell, a Makefile typo, or a CI matrix that inherits a developer's
environment can flip the trigger by accident. The allow-gate is the
single switch CI pins OFF, so even a leaked trigger is harmless.

Only the literal value `"1"` unlocks the gate. `"true"`, `"yes"`,
`"y"` are all treated as OFF — typos cannot accidentally enable it.

### 1a. Determinism pre-check (third key)

The two env-var keys above protect against *accidental* rewrites.
They do **not** protect against rewriting fixtures with
*non-deterministic bytes* (map-iteration order, embedded
timestamps, locale-dependent floats, randomized IDs). A flaky
writer would pass pass-1 (env vars satisfied) but fail pass-2
(verify) — after the now-bad bytes have already landed on disk.

Use `goldenguard.AllowUpdateAfterDeterminism` instead of
`goldenguard.AllowUpdate` in any new golden test. It runs the
writer 3 times in-process and refuses to consult the env-var gate
unless every run produced byte-identical output:

```go
if goldenguard.AllowUpdateAfterDeterminism(t, trigger,
    "clonefrom_report_canonical.json",
    func() ([]byte, error) { return summary.MarshalReport(rows) },
) {
    os.WriteFile(path, got, 0o644)
    return
}
```

Existing tests using `goldenguard.AllowUpdate(t, trigger)` still
work — adopt the new helper opportunistically when touching a
golden test, or eagerly for any writer that has historically been
flaky.

For a pure determinism assertion (no rewrite intent),
`goldenguard.AssertWriterDeterministic(t, label, writer)` is
available and safe to call in regular CI runs.

## 2. Standard Regeneration Command

```bash
GITMAP_UPDATE_GOLDEN=1 GITMAP_ALLOW_GOLDEN_UPDATE=1 \
  go test ./gitmap/<package>/ -run <TestNameOrPattern>
```

**Always scope with `-run`** to the specific test(s) whose fixtures
you intend to rewrite. A bare `go test ./...` with both env vars set
will rewrite every golden in the repo — including ones you did not
mean to touch.

### Two-pass verification (mandatory)

After regenerating, **re-run the same tests without the env vars**:

```bash
go test ./gitmap/<package>/ -run <TestNameOrPattern>
```

This second pass must pass cleanly. It confirms the regenerated bytes
are stable (i.e. the writer is deterministic) and that the new fixture
matches what the production code actually emits.

If the second pass fails, your writer is non-deterministic (map
iteration order, timestamps, locale-dependent floats) — fix the
writer, do not re-regenerate.

## 3. Worked Examples

### 3a. Clone-from JSON report

Files: `gitmap/clonefrom/testdata/clonefrom_report_*.json`
Tests: `TestCloneFromReportJSON_Golden_Empty`,
`TestCloneFromReportJSON_Golden_Canonical`.

```bash
GITMAP_UPDATE_GOLDEN=1 GITMAP_ALLOW_GOLDEN_UPDATE=1 \
  go test ./gitmap/clonefrom/ -run TestCloneFromReportJSON_Golden

go test ./gitmap/clonefrom/ -run TestCloneFromReportJSON_Golden
```

Files to commit:
- `gitmap/clonefrom/summary.go` (the writer change)
- `gitmap/clonefrom/testdata/clonefrom_report_empty.json`
- `gitmap/clonefrom/testdata/clonefrom_report_canonical.json`
- `CHANGELOG.md` (downstream-visible break note — see §5)

If the change involves the `schemaVersion` field, **also** bump the
literal in `gitmap/clonefrom/summary_jsonschemaversion_test.go`
(`expectedSchemaVersionPinned`) and the constant
`constants.CloneFromReportSchemaVersion` in the same commit.

### 3b. Clone-from CSV report (including quoting fixtures)

Files: `gitmap/clonefrom/testdata/clonefrom_report_*.csv`

```bash
GITMAP_UPDATE_GOLDEN=1 GITMAP_ALLOW_GOLDEN_UPDATE=1 \
  go test ./gitmap/clonefrom/ -run 'TestCloneFromReportCSV_Golden|TestCloneFromReportCSV_Quoting'

go test ./gitmap/clonefrom/ -run 'TestCloneFromReportCSV_Golden|TestCloneFromReportCSV_Quoting'
```

### 3c. Scan / clone-now / E2E plan goldens

```bash
# Scan formatter goldens
GITMAP_UPDATE_GOLDEN=1 GITMAP_ALLOW_GOLDEN_UPDATE=1 \
  go test ./gitmap/formatter/ -run ScanGolden

# Clone-now cross-format consistency
GITMAP_UPDATE_GOLDEN=1 GITMAP_ALLOW_GOLDEN_UPDATE=1 \
  go test ./gitmap/clonenow/ -run TestCloneNow_CrossFormat

# Scan→clone-from end-to-end plan
GITMAP_UPDATE_GOLDEN=1 GITMAP_ALLOW_GOLDEN_UPDATE=1 \
  go test ./gitmap/tests/scanclone_test/ -run TestScanClone_E2E
```

### 3d. CLI command stream tests (`cmd` package)

The `cmd` package uses an additional `-update` flag on top of the
two env vars. Pattern:

```bash
GITMAP_ALLOW_GOLDEN_UPDATE=1 \
  go test ./gitmap/cmd/ -run TestCloneStream -update
```

## 4. Files to Commit

Every regenerate PR must include, in a **single commit**:

1. The writer/code change that caused the format drift.
2. Every regenerated file under `testdata/` — never partial.
3. Any pinned literal that mirrors the format (e.g.
   `expectedSchemaVersionPinned`, `CloneFromReportSchemaVersion`).
4. A `CHANGELOG.md` entry under the next release if the change is
   visible to downstream tooling (see §5).

**Never** commit a fixture rewrite without the corresponding code
change — reviewers cannot tell intentional drift from accidental
drift in that case.

**Never** rewrite a fixture by hand-editing the file. Always go
through the test-driven regenerate path so the fixture matches what
the production writer actually produces.

---

## 5. CHANGELOG Requirements

A regenerate is a downstream-visible event when **any** of these are
true:

- A column or JSON field was added, removed, renamed, or reordered.
- A schema-version constant was bumped.
- A status enum string changed (`ok` → `success`, etc.).
- The empty-case shape changed (e.g. `[]` → `null`, or vice versa).
- The numeric format of a field changed (e.g. integer → float).
- The CSV line ending or quoting policy changed.

For any of the above, add a `### Breaking` or `### Changed` entry to
`CHANGELOG.md` describing the consumer impact and the migration path
(e.g. "JSON report now wraps rows in `{schemaVersion, rows}` —
update jq to `.rows[]`").

Pure value changes (a new repo URL appearing in a fixture, a refreshed
duration measurement) do **not** require a CHANGELOG entry.

---

## 6. CI Posture

CI configurations (GitHub Actions, local `make` targets, contributor
scripts) **must not** export `GITMAP_ALLOW_GOLDEN_UPDATE` under any
circumstance. The variable's only purpose is to be unset in every
non-developer context.

If a CI job needs to verify regeneration logic itself (rare —
typically only the `goldenguard` package's own tests), it must scope
the env var to a single `go test` invocation with `env -i` or
explicit `-run` filters and unset it immediately after.

## 7. Troubleshooting

| Symptom                                                    | Cause                                                    | Fix                                                                  |
|------------------------------------------------------------|----------------------------------------------------------|----------------------------------------------------------------------|
| `t.Fatalf: golden update requested but GITMAP_ALLOW_GOLDEN_UPDATE is not set to "1"` | Trigger set, allow-gate missing                           | Export `GITMAP_ALLOW_GOLDEN_UPDATE=1` for the same shell             |
| Second-pass verification still fails after regenerate     | Writer is non-deterministic                              | Fix the writer (sort maps, drop timestamps); do not regenerate again |
| Fixture diff includes unrelated files                     | Forgot to scope with `-run`                              | `git checkout -- <unrelated>` then re-run with a tighter `-run`      |
| `schemaVersion` test fails after regenerate               | Bumped writer/golden but forgot the pinned literal       | Update `expectedSchemaVersionPinned` to match the new constant       |
| CI green locally but red in pipeline                      | Local shell has `GITMAP_UPDATE_GOLDEN=1` exported globally | `unset GITMAP_UPDATE_GOLDEN GITMAP_ALLOW_GOLDEN_UPDATE` in `.zshrc`  |
